# 共有前チェックリスト

このチェックリストは、WebアプリのURLやGitHubリポジトリを共有する前に、アプリの動作、セキュリティ、個人情報、機密情報の扱いに問題がないかを確認するためのものです。

---

## 1. 本番環境の確認

想定本番構成である Render と Supabase を使ったデモ環境が、正しく動いていて、外部からアクセスできる状態になっているかを確認します。リポジトリだけでは現在の稼働状態を判断できないため、確認日も記録します。

> [!NOTE]
> **Render Freeプランに関する注意**
> 想定本番構成で Render Freeプランを使用している場合、しばらくアクセスがないと、初回アクセス時に起動まで時間がかかることがあります。

- [ ] **Renderの最新デプロイ確認**: Renderのダッシュボードで、最新のデプロイが「Deployed」と表示され、正常に完了しているか。
- [ ] **反映されているブランチの確認**: Renderでデプロイ対象になっているブランチ（通常は `main`）が、GitHub上の最新コードと一致しているか。
- [ ] **シークレットウィンドウでの表示確認**: ブラウザのシークレットウィンドウ（プライベートブラウズ）で本番環境URLを開き、キャッシュやCookieの影響を受けずに表示できるか。
- [ ] **ログイン画面の表示確認**: ログインしていない状態でアクセスしたときに、正しくログイン画面が表示されるか。
- [ ] **デモユーザーでのログイン確認**: 事前に用意したデモユーザー（人事、店長、一般従業員）で、正しくログインできるか。
- [ ] **主要画面の表示確認**: ログイン後、ダッシュボード、シフト管理、勤怠画面などを開いたときに、500エラーなどのサーバーエラーが出ず、正常に表示されるか。
- [ ] **デモデータの確認**: カレンダーや一覧画面に、動作確認しやすい量のデモデータが表示されているか。

---

## 2. 主要機能の確認

人事、店長、一般従業員それぞれの立場で、必要な画面や操作が想定どおりに使えるかを確認します。

### ロール別ログイン・画面表示

- [ ] **人事（HR）ユーザー**: 人事アカウントでログインし、全社シフト、従業員管理、マスタ管理などの画面が表示されるか。
- [ ] **店舗責任者（店長/MANAGER）ユーザー**: 店長アカウントでログインし、原則として自分と同じ拠点・部署に所属するスタッフのシフト作成や申請承認の画面が表示されるか。
- [ ] **一般従業員（EMPLOYEE）ユーザー**: 一般従業員アカウントでログインし、自分のシフト確認や出退勤打刻の画面が表示されるか。

### 勤怠・打刻フロー

- [ ] **出退勤の打刻**: 従業員ユーザーが「出勤」「退勤」ボタンを押し、現在時刻が勤怠実績に正しく反映されるか。
- [ ] **勤怠修正申請の作成**: 従業員ユーザーが「打刻修正」画面から、打刻漏れなどに対する修正申請を作成できるか。
- [ ] **修正申請の承認**: 自己承認が禁止され、店長は担当範囲内、人事は権限規則に従って、届いた打刻修正申請を承認できるか。設定された代理店長が処理できる場合も確認する。
- [ ] **修正申請の却下**: 権限を持つ担当者が届いた打刻修正申請を却下できるか。却下理由を入力し、申請者への通知および操作履歴に記録できるか。
- [ ] **承認後の反映確認**: 申請が承認されたあと、対象の勤怠データが申請された時間に変更されているか。
- [ ] **却下後の確認**: 申請が却下された場合、対象の勤怠データが変更されず、元の状態のままになっているか。

### シフト・通知・勤怠管理

- [ ] **シフト管理画面の表示**: カレンダーを使ったシフト調整画面が正常に表示されるか。
- [ ] **他拠点シフトの閲覧範囲**: 一般従業員がヘルプ調整用に他拠点の確定済み月間シフト表を閲覧でき、社員番号や詳細メモが表示されないか。また、この閲覧画面から編集・削除・承認ができないか。
- [ ] **通知画面の表示**: ホーム画面や通知一覧画面に、新着のお知らせが正しく表示されるか。
- [ ] **通知リンクの確認**: 通知に表示されているリンクを押したときに、対象の申請画面や管理画面へ移動できるか。
- [ ] **月次勤怠データの確認**: 従業員の「勤怠実績」や店長の「月次確定」画面で、月ごとの勤怠データを一覧で確認できるか。
- [ ] **確定後の操作制限**: 確定された勤怠実績について、確定解除されるまで打刻変更や修正申請ができないか。
- [ ] **権限に応じた表示確認**: ログインしているユーザーの権限に合わせて、サイドメニューや操作ボタンの表示が切り替わっているか。

## 3. セキュリティ確認

ログイン状態やユーザーの権限に応じて、見られる画面・操作できる範囲が正しく制限されているかを確認します。

- [ ] **未ログイン時のアクセス制限**: ログインしていない状態で保護された画面（`/app/*`）へ直接アクセスしたときに、ログイン画面へ戻されるか。
- [ ] **人事専用画面のアクセス制限**: 一般従業員や店長が、人事専用の画面（従業員管理、マスタ管理など）へ直接アクセスした場合、403 Forbidden などのアクセス拒否画面が表示されるか。
- [ ] **店長専用画面のアクセス制限**: 一般従業員が、店長専用の画面（シフト調整、勤怠管理など）へ直接アクセスした場合、アクセスが制限されるか。
- [ ] **管理対象外データの操作制限**: 店長が、原則として自分と同じ拠点・部署以外の従業員の申請を操作できないようになっているか。
- [ ] **他人のデータ閲覧・操作制限**: 一般従業員がURLのIDなどを変更しても、他人の勤怠情報や申請履歴を見たり操作したりできないようになっているか。
- [ ] **操作ボタンの表示制御**: 勤怠修正の「承認」「却下」ボタンが、権限規則上その申請を処理できる店長・人事・代理店長にだけ表示されるか。自己承認用のボタンが表示されないか。
- [ ] **二重操作の防止**: すでに承認済み、または却下済みの申請に対して、再度承認・却下ができないようになっているか。
- [ ] **ログアウト後の保護**: ログアウト後、ブラウザの「戻る」ボタンを押しても、保護された画面で操作できないようになっているか。
- [ ] **エラー画面の情報保護**: システムエラーや404エラーが発生した場合でも、エラー画面にデータベース接続情報や詳しいエラー内容が表示されないようになっているか。
- [ ] **パスワードの保護**: ユーザーのパスワードが画面、ログファイル、GitHubの履歴などにそのまま表示・記録されていないか。

## 4. 個人情報・デモデータの確認

氏名、メールアドレス、申請理由、通知内容などに、個人情報や不適切な内容が含まれていないかを確認します。

- [ ] **架空の氏名・メールアドレスの確認**: 登録されているユーザー名やメールアドレスが、確認用の架空データになっているか。例：`@example.com` など。
- [ ] **電話番号・住所の確認**: データベースや画面上に、実在する個人の電話番号や住所が登録されていないか。
- [ ] **社員番号の確認**: 社員番号などに、実在する個人を特定できる番号やコードが使われていないか。
- [ ] **過去の業務情報の排除**: 過去の職場、学校、顧客、プロジェクト名など、外部に出せない情報が残っていないか。
- [ ] **デモデータとしての適切性**: 申請理由や通知文面に、個人的すぎる内容や不適切な表現が含まれていないか。
- [ ] **データリセット方法の確認**: 必要に応じて、本番デモ環境のデータを初期状態に戻す手順や設定が用意されているか。

---

## 5. GitHub上の機密情報確認

公開するGitHubリポジトリに、パスワードや接続情報などの大切な情報が誤って含まれていないかを確認します。

- [ ] **環境変数ファイルの除外**: `.env` などの設定ファイルが `.gitignore` に指定され、GitHubに上がらないようになっているか。
- [ ] **ソースコード内のパスワード確認**: `DB_PASSWORD` やAPIキーなどの認証情報が、JavaコードやREADMEに直接書かれていないか。
- [ ] **JDBC接続URLの確認**: Supabaseなどの本番データベースに接続するための `JDBC_URL` に、実際のパスワードが含まれていないか。
- [ ] **Supabase接続情報の確認**: SupabaseのプロジェクトURLやAPIキーなどが、フロントエンドやバックエンドのソースコードに直接書かれていないか。
- [ ] **Render環境変数の確認**: Renderの管理画面で設定する環境変数の実際の値が、ソースコード内にそのまま書かれていないか。
- [ ] **初期管理者のログイン情報確認**: 初期セットアップで作成される人事ユーザーの本物のメールアドレスや初期パスワードが、コードやドキュメントに残っていないか。
- [ ] **テストコード・ログの確認**: テスト用コードや過去のログファイルの中に、本番環境のパスワードや接続情報が残っていないか。
- [ ] **スクリーンショットの確認**: READMEやドキュメントに載せている画像に、機密情報、個人情報、ブラウザのタブ名、URLなどが写り込んでいないか。

---

## 6. 共有前の最終確認

URL、説明文、ログイン情報、関連ドキュメントの案内が揃っているかを最後に確認します。

- [ ] **アプリURLの確認**: 共有するWebアプリのURLが正しいか。
- [ ] **GitHubリポジトリURLの確認**: 共有するGitHubリポジトリのURLが正しいか。また、リポジトリが「Public（公開）」に設定されているか。
- [ ] **デモアカウント情報の整理**:
  - 人事（HR）、店長（MANAGER）、一般従業員（EMPLOYEE）それぞれのログインに必要なメールアドレスとパスワードが、見やすく整理されているか。
  - ※実際のデモ用ログイン情報は、共有時に別途記載します。
- [ ] **READMEの内容確認**: アプリの目的、使用技術、起動方法、デモ環境の案内などがREADMEにわかりやすく記載されているか。
- [ ] **関連ドキュメントへの案内確認**: 関連ドキュメントへのリンクが、READMEのドキュメント一覧から確認できるか。
- [ ] **見てほしい機能の整理**: 権限ごとの画面制御、Supabase接続、勤怠・シフト調整の整合性など、ポートフォリオとして伝えたい工夫がわかりやすく整理されているか。
- [ ] **起動遅延の説明確認**: Render Freeプランのため、初回アクセス時に起動まで時間がかかることがある旨をREADMEに記載しているか。
- [ ] **公開後の安全対応**: 必要に応じてデモユーザーのパスワードを変更したり、デモ環境を停止・削除したりする手順を把握しているか。
