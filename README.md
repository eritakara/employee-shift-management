# シフト・有休・勤怠管理アプリ

このアプリは、シフト調整、有休申請、および毎日の出退勤打刻・勤怠確認を Web 上で一元管理できるアプリケーションです。
実務の業務システムに近いデータや運用フローを想定しており、本番環境も用意しています。

---

## 環境

ポートフォリオとして実際に操作を確認していただけるよう、コンテナ環境およびクラウドデータベースを用いた本番環境を公開しています。

* **URL**: [https://employee-shift-management-demo.onrender.com/](https://employee-shift-management-demo.onrender.com/)
  * *※ Render の無料プランを使用しているため、しばらくアクセスがないと自動的にスリープ状態になります。初回アクセス時は起動するまで30秒〜60秒程度お時間がかかります。*
* **データベース**: Supabase (PostgreSQL)
* **ホスティング**: Render (Dockerコンテナによるデプロイ)

## 主な機能

このアプリでは、ログインするアカウントの権限（人事、店長、一般従業員）に応じて、表示画面や操作可能な機能が動的に切り替わります。

### 1. 共通機能
* **ログイン / ログアウト**: 各ロールのアカウントでログインし、それぞれの専用画面へ遷移します。
* **通知表示**: 申請の承認状況や、店長からのシフト提出依頼などをホーム画面で素早く確認できます。

### 2. 人事管理者（HRロール）の機能
* **全社・支店ごとのシフト状況確認**: 全拠点のシフト・勤務状況を確認できます。
* **従業員アカウント管理**: 新規アカウントの登録、権限設定、所属変更、有効/無効の切り替えなどが可能です。
* **マスタデータ管理**: 営業所（拠点）、部署、雇用形態、勤務区分などの追加・編集が行えます。

### 3. 店舗責任者（店長 / MANAGERロール）の機能
* **自店舗のシフト作成・編集**: カレンダーUIを用いて、所属従業員のシフトを個別に、または一括で作成・編集できます。※要確認
* **従業員からの申請承認**: 提出された希望シフトや有休休暇の申請内容を確認し、承認・却下を行います。
* **出勤簿・打刻状況の確認**: 自店舗の従業員の打刻データ、遅刻・早退・残業・退勤未打刻といったアラート状況を一覧で確認します。

### 4. 一般従業員（EMPLOYEEロール）の機能
* **確定シフトの確認**: 自身の確定したシフトを月間カレンダー表示、各支店ごとにリスト形式で確認できます。
* **希望シフトの提出**: 来月以降の勤務希望日や時間帯を入力し、店長に提出できます。
* **有休の申請**: 取得区分（全休・半休・時間休）を入力して店長に有休申請します。申請の進捗状況も随時追跡可能です。
* **出退勤打刻**: 出勤時・退勤時にワンクリックで現在時刻を打刻します。
* **個人出勤簿の確認**: 自身の打刻履歴、労働時間、遅刻・早退などの有無を確認できます。

## テストデータ

実務に近いデータを本番データベース内に事前に構築しています。

* **アカウント**: 人事管理者、各拠点の店長、一般従業員を含む **合計35名分**
* **シフトデータ**:
  * 2026年5月分：確定済みのシフトデータ（35名分）
  * 2026年6月分：確定済みのシフトデータ（35名分）
  * 2026年7月・8月分：各従業員から提出された「希望シフト」データ（店長画面から承認フローをテストできます）
* **勤怠打刻データ**: 2026年5月分の出退勤打刻実績データ（遅刻や残業、退勤時の打刻漏れなどのデータを含み、店長画面のアラート表示や出勤簿で確認できます）
* **有休申請データ**: 2026年7月〜9月分の有休申請データ（承認待ち、承認済み、却下などのステータスを含み、承認プロセスを試せます）

## 開発において工夫した点

1. **ロール（権限）による厳密な機能・画面制御**
   * ログインユーザーの権限をチェックしています。自身の権限以外でアクセスした場合でも、安全に権限エラー画面へリダイレクトされるセキュアな設計にしました。
2. **Render + Supabase PostgreSQL での公開環境構築**
   * 無料枠の制約（IPv6非対応など）に起因する接続不良を解消するため、Supabase の Session Connection Pooler を経由した JDBC 接続を採用し、安定した本番公開環境を構築しました。
3. **HikariCPによるDB接続の効率化**
   * SQL実行ごとの接続生成コストを抑えるため、HikariCPによるDBコネクションプールを導入しました。Render + Supabaseの省リソース環境に合わせて `maximumPoolSize=3`、`minimumIdle=0` に調整し、ダッシュボード・有休・勤怠・シフトなどの画面遷移を改善しました。
   * 本番環境でHR・店長・一般従業員の主要画面を確認し、500エラーがないことと、導入前より画面遷移が軽くなったことを確認しています。

---

## 開発で苦労した点と解決策

1. **Supabase (PostgreSQL) 接続における環境変数設定と IPv4/v6 到達性問題**
   * **課題**: Render の無料コンテナインスタンスは IPv6 に対応しておらず、Supabase の直接接続用ホスト名に接続しようとすると `Network is unreachable` エラーが発生しました。
   * **対策**: Supabase 側で Session Pooler (IPv4対応) を有効にし、JDBC URL や環境変数（`DB_USER` や `DB_PASSWORD`）をプーラー用に厳密に切り分けることで安定した接続を確立しました。
2. **PostgreSQL と H2 データベースにおける SQL 記法の差異**
   * **課題**: 両者で互換性のない SQL 記法（日付処理や文字列結合など）が原因でエラーが発生しました。
   * **対策**: 標準 SQL に基づいたクエリの記述を徹底し、接続ドライバごとの挙動を意識した SQL 設計に変更して両方の環境でコードが動作するように対応しました。
3. **シフト、勤怠、有休申請におけるデータの整合性制御**
   * **課題**: 同一の日に「シフトが入っていないのに出勤打刻ができる」「有休を取得している日にシフトが重複して入ってしまう」といったデータ整合性の崩れを防ぐ必要がありました。
   * **対策**: 申請登録およびシフト作成において、日付の重複や整合性を検証するバリデーションを強化し、業務ルールに即したエラーメッセージをフロントエンドへ返す仕組みを実装しました。
