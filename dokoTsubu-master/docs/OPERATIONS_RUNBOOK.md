# シフト・有休管理 障害・問い合わせ・アカウント復旧ランブック

## 1. 連絡と優先度

| 重要度 | 判断基準 | 初動目標 | 例 |
|---|---|---|---|
| P1 緊急 | 全員利用不可、データ破損・漏えい疑い、打刻不能が広範囲 | 15分以内に責任者へ連絡し、変更作業を止める | HTTP停止、DB接続不能、不正アクセス疑い |
| P2 高 | 一部の主要機能・所属が利用不可、メール滞留、日次処理失敗 | 1時間以内に担当者を決定 | 承認不能、出力不能、SMTP障害 |
| P3 通常 | 回避策がある個別不具合、操作質問 | 1営業日以内に受付回答 | 表示崩れ、利用方法、個別アカウント |

受付記録には、発生日時、申告者、環境、画面、操作、期待結果、実際の結果、影響人数、再現性を記載する。パスワード、リセットトークン、位置情報、申請理由、DBファイル、ログ全文をチケットやメールへ添付しない。

## 2. 共通初動

1. 本番・ステージング・開発のどの環境かを確認する。
2. P1の場合は新規デプロイ、データ移行、手動SQLを中止し、業務責任者とシステム責任者へ連絡する。
3. 次の読取専用診断を実行する。

```powershell
powershell -ExecutionPolicy Bypass -File .\ops\check-health.ps1 `
  -Url https://shiftflow.example/shiftflow/ `
  -TomcatBase C:\ShiftFlow\production\tomcat
```

4. 診断結果のHTTP状態、DB存在・容量、ログ更新日時だけを記録する。ログ本文はアクセス権のある担当者がサーバー内で確認する。
5. 発生直前のデプロイ、設定変更、証明書更新、ディスク使用量、OS・Tomcat再起動を確認する。
6. 調査担当、判断担当、次回連絡時刻を記録する。

## 3. 障害別手順

### アプリへ接続できない

1. DNS・TLS・リバースプロキシの応答とTomcatプロセスを確認する。
2. Tomcatログの直近時刻をサーバー内で確認する。ログを外部へ転送する場合は個人情報をマスキングする。
3. DBファイルの存在、ディスク空き容量、サービスアカウントのアクセス権を確認する。
4. 原因不明のままDBファイルを移動・コピー・削除しない。
5. 再起動は責任者の承認後に一度だけ実施し、実施時刻と結果を記録する。再発時は再起動を繰り返さずP1として調査する。

### DB異常・データ不整合の疑い

1. 書込みを増やす操作、手動修正、デプロイを停止する。
2. 接続可能ならH2 `SCRIPT` バックアップを取得する。稼働中の `.mv.db` を直接コピーしない。
3. [バックアップ・復元手順](../ops/BACKUP.md)で空の別DBへ復元する。本番DBを復元先に指定しない。
4. 元DBと復元検証DBの件数・対象業務データを照合し、差異を記録する。
5. 本番切替・データ修正は、業務責任者とシステム責任者の承認後に別の作業計画として実施する。

### メールが届かない

1. 画面のメール送信履歴で `QUEUED`、`RETRY`、`FAILED`、`SENT` を確認する。
2. SMTPホスト、送信元、TLS、認証情報の設定有無を確認する。秘密値を画面共有・チケットへ貼らない。
3. `RETRY` は1分、5分、15分間隔で再試行され、設定回数到達後に `FAILED` となる。
4. SMTP復旧後、人事担当者が失敗メールの再送操作を行う。重複送信を避けるため同じメールを別手段で並行送信しない。
5. `last_error` は認証関連文字列がマスキングされるが、そのまま外部共有しない。

### 日次処理・通知の異常

1. Tomcatが0:05に稼働していたか、`Daily task failed` が記録されていないか確認する。
2. 希望シフト通知は締切前日の未提出者だけが対象で、同日重複送信されない仕様を確認する。
3. 有休処理、保存期間処理、期限切れトークン処理への影響範囲を確認する。
4. 手動再実行は重複影響を評価し、システム責任者の承認を得てから行う。

## 4. アカウント復旧

### 本人確認

1. 申告者の氏名、従業員番号、登録メールアドレスを社内の承認済み本人確認方法で照合する。
2. パスワードを聞かない。本人確認資料を通常メールやチケットへ添付させない。
3. 退職・休職・無効化対象でないことを人事担当者が確認する。

### パスワードを忘れた場合

1. 利用者にログイン画面の「パスワードを忘れた」から登録メールアドレスを入力してもらう。
2. 存在する・しないにかかわらず同じ受付表示となるため、画面表示だけで登録有無を回答しない。
3. 再設定リンクは24時間有効で、一度使用すると無効になる。新規発行時は以前の未使用リンクも無効になる。
4. メール障害時は先にSMTPを復旧し、再発行する。トークンを担当者が読み上げたり、チャットへ貼ったりしない。
5. 完了後、`ISSUE_RESET_TOKEN` と `CONSUME_RESET_TOKEN` の監査記録を確認する。パスワードハッシュやトークン値は監査へ記録しない。

### ログイン試行上限の場合

同じ接続元・メールアドレスで5分以内に5回失敗すると、5分間ログインを受け付けない。本人確認後も手動でDBを変更せず、5分経過を待って正しいパスワードまたは再設定後のパスワードで再試行する。

### メールアドレス不明・アカウント無効の場合

本人確認後、人事担当者が従業員台帳と登録情報を照合する。無効アカウントを復旧する場合は、人事承認、理由、適用日時を記録し、通常の従業員情報変更手順を使う。DBを直接更新しない。

## 5. 問い合わせ処理

1. 操作質問、データ訂正、障害、権限変更、機能要望に分類する。
2. 権限変更・所属変更・退職者復帰は人事承認を必須とする。
3. 勤怠や有休の訂正は画面の申請・承認機能を使用し、監査履歴を残す。
4. 回答には対象環境、確認した仕様、利用者が行う次の操作を記載する。
5. 不具合の場合は再現手順を最小化し、実データをテスト環境へ持ち込まない。
6. 解決時に原因、対応、影響、再発防止、利用者への回答日時を記録して終了する。

## 6. 復旧完了判定

- ヘルスチェックが合格する。
- 従業員、店長、人事担当者の代表アカウントで権限別の主要画面を確認する。
- DB件数と対象業務データに想定外の差異がない。
- メール・日次処理に滞留や重複がない。
- 監査ログに対応操作が残り、秘密情報・位置情報・申請理由が漏れていない。
- 業務責任者とシステム責任者がサービス再開を承認する。
