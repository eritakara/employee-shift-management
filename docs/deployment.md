# インフラ・デプロイ手順書

本ドキュメントは、本アプリケーションをローカル環境で起動する手順、および本番環境（Render / Supabase）へ配備（デプロイ）するための設定とインフラ構成について解説したものです。

---

## 1. インフラ構成の概要

本アプリケーションは、コンテナ技術を用いた軽量なWebコンテナサーバーと、マネージドなクラウドデータベースを組み合わせたモダンな構成で稼働しています。

* **Webアプリケーション実行環境**: **Render Web Service**
  * Dockerコンテナを用いて Tomcat 10.1 および Java 21 の動作環境を起動し、WARファイルをルート配下に配備（コンテキストパス `/` で稼働）させています。
* **データベース環境**: **Supabase PostgreSQL**
  * クラウド上に設置された PostgreSQL データベースを利用し、データの一元管理を行っています。

---

## 2. Supabaseへの接続と Session Pooler の採用

Render の実行サーバーから Supabase PostgreSQL へ安全かつ安定して接続するために、**Supabase Session Pooler** 経由のJDBC接続を採用しています。

### Session Pooler 経由で接続する理由
1. **リソース制約の解消**: Renderの無料プラン等から接続する際、クラウドDBへの直接接続で到達性エラーが発生するケースを防ぎます。
2. **接続プールの安定化**: Session Pooler（ポート5432）を経由することで、Tomcat Webコンテナ内の接続プール（HikariCP）によるデータベースコネクションの破棄と再接続を安定して行うことが可能になります。
   * ※注意: プーラーの動作モードは必ず **Session** モードに指定する必要があります（Transactionモードは JDBC の Prepared Statement と競合して動作しないため利用不可）。

## 3. デプロイに必要な環境変数

本番環境（Renderのダッシュボード設定等）で定義する環境変数は以下の通りです。セキュリティ保護のため、パスワードや接続情報の「実値」は公開リポジトリにはコミットせず、管理コンソール側で安全に設定します。

| 環境変数名 | 設定内容の例 / 説明 |
| :--- | :--- |
| **`APP_ENV`** | `production`（本番環境であることを明示。起動エラーのチェック処理を有効にします。） |
| **`JDBC_URL`** | `jdbc:postgresql://[ホスト名]:5432/postgres?sslmode=require` （SupabaseのSession Pooler用接続URL。※ユーザー名とパスワードは含めません） |
| **`DB_USER`** | 接続用のユーザー名（SupabaseのSession Pooler設定画面に表示されるユーザー名を指定） |
| **`DB_PASSWORD`** | データベース接続用の暗号化されたパスワード値 |
| **`INITIAL_HR_EMAIL`** | 本番稼働時に最初に作成される初期管理者（人事アカウント）のメールアドレス |
| **`INITIAL_HR_PASSWORD`** | 初期管理者の初期ログインパスワード（デフォルト値のまま起動しようとすると、安全のため起動エラーになります） |
| **`DEMO_ACCOUNTS_ENABLED`** | `true` （ポートフォリオデモ環境用に各拠点のデモアカウントを自動生成する場合に設定） |
| **`DEMO_ACCOUNTS_PASSWORD`** | 自動生成されるデモアカウントの共通パスワード（セキュリティ保護のため、初期値以外のパスワード値の指定が必須） |

## 4. Docker / Tomcatの役割

### 4.1 Dockerfile によるパッケージング
本プロジェクトに同梱されている `Dockerfile` は、マルチステージビルドを使用しています。
1. **ビルドステージ**: Maven等の構成と同様に、ソースコードをコンパイルし、Webアプリケーションのアーカイブファイル（`ROOT.war`）を生成します。この段階で、本番接続用の PostgreSQL JDBC ドライバーが自動ダウンロードされ同梱されます。
2. **実行ステージ**: 公式の Tomcat 10.1 (JDK 21) イメージをベースに、生成された `ROOT.war` を `webapps` ディレクトリ配下にコピーします。これにより、環境依存を排除した同一のビルド結果がローカルDockerおよびRender上のコンテナ内で再現されます。

## 5. ローカル環境での起動手順

ローカルでの開発・動作確認には、以下の2つの手順が用意されています。

### 5.1 PowerShell スクリプトによるローカル Tomcat 起動
ローカル環境に Java 21 および Tomcat 10.1 がインストールされている場合、以下のスクリプトを実行することでビルドと起動が自動で行われます。

```powershell
# 1. アプリケーションのビルド
powershell -ExecutionPolicy Bypass -File .\build.ps1

# 2. ローカルTomcat上での起動 (既定では H2 データベースを使用)
powershell -ExecutionPolicy Bypass -File .\run-dev.ps1
```
起動完了後、ブラウザで `http://localhost:8080/` にアクセスします。

### 5.2 ローカル Docker による起動
Tomcatを直接ローカルに用意せず、コンテナとして動作させる場合の手順です。

```powershell
# 1. コンテナイメージのビルド
docker build -t shiftflow .

# 2. ポート10000でコンテナを起動
docker run --rm -p 10000:10000 shiftflow
```
起動完了後、ブラウザで `http://localhost:10000/` にアクセスします。

## 6. 本番環境での注意事項とトラブルシューティング

### 6.1 Render Freeプランによる起動遅延について
Renderの無料プラン（Free Plan）を使用しているため、一定時間アクセスがない場合、コンテナサーバーが自動的に休止状態（スリープ）に入ります。休止後の最初のアクセス時には、コンテナの起動処理（Tomcatの初期化とDB接続検証）が走るため、**画面が表示されるまでにしばらく時間がかかることがあります**。
ポートフォリオの確認を行う際は、最初の読み込み時に待ち時間が発生することをあらかじめご了承いただき、画面が表示されるまでお待ちください。

### 6.2 起動時エラー（ログに `IllegalStateException` が出力される場合）
* **原因**: 初期管理者のパスワード（`INITIAL_HR_PASSWORD`）やデモ用パスワードが初期値のままであるか、設定された `JDBC_URL` に不備がある場合に、システムの自己診断機能が作動してアプリの起動を強制停止しています。
* **対策**: Renderの環境変数設定を見直し、安全なパスワード値が個別に設定されているかを確認のうえ、再起動（Clear Cache & Deploy）を実行してください。
