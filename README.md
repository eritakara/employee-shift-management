# 従業員シフト・勤怠管理システム

スタッフのシフト作成・調整と、日々の出退勤記録をWeb上でまとめて管理できるアプリケーションです。
店舗や事業所での利用を想定し、人事・店長・一般従業員など、それぞれの役割に合わせて使える機能が変わるようにしています。

---

## 使用技術・環境

* **プログラム言語**: Java 21
* **フレームワーク/技術**: Java Servlet / JSP (Tomcat 10.1 / Jakarta EE 10)
* **フロントエンド**: HTML5, CSS (Vanilla CSS), JavaScript
* **データベース**: Supabase PostgreSQL (本番用) / H2 Database (ローカル開発・テスト用)
* **DB接続プール**: HikariCP
* **インフラ / デプロイ**: Render (Dockerコンテナによるデプロイ)

## 主な機能

ログインしたユーザーの役割（人事、店長、一般従業員）に応じて、表示される画面や操作できる内容が自動的に切り替わります。

### 1. 権限管理・ログイン機能
* **ログイン/ログアウト**: 役割ごとのアカウントでログインし、それぞれの専用画面を利用できます。ログアウト後は、再度ログインしないと操作できないようにしています。
* **アクセス制御**: 従業員が管理画面へ直接アクセスしようとした場合や、店長が原則として自分と同じ拠点・部署以外のデータを操作しようとした場合は、サーバー側でアクセスを制限します。

### 2. 出退勤打刻機能
* **Web打刻機能**: 従業員は、勤務開始時と終了時にワンクリックで打刻できます。
* **遅刻・早退の自動判定**: 確定したシフトと実際の打刻時刻を照らし合わせ、遅刻や早退を自動で判定・記録します。

### 3. シフト作成・調整機能
* **シフト作成・編集**: 店長や人事が、カレンダー画面などを使ってスタッフのシフトを作成・調整できます。
* **シフト確認**: 欠勤時などに拠点間のヘルプ調整を行いやすくするため、一般従業員と店長は、他拠点を含む拠点ごとの確定済み月間シフト表を閲覧できます。閲覧用の表には氏名、勤務日、勤務区分など必要最小限の情報だけを表示し、社員番号や詳細メモは表示しません。この画面から他拠点のシフトを編集・削除・承認することはできません。

### 4. 勤怠修正申請の作成と承認・却下フロー
* **修正申請の作成**: 従業員は、打刻漏れや打刻ミスがあった場合に、修正したい時刻と理由を入力して申請できます。
* **承認・却下機能**: 店長は担当範囲内の申請を、人事は権限規則に従って申請を処理します。自己承認はできません。設定された代理店長が処理できる場合もあります。承認された場合は勤怠実績に反映され、却下された場合は元のデータがそのまま残ります。

### 5. 通知機能
* **申請と処理の通知**: 修正申請が届いたときや、申請が承認・却下されたときに、システム内のダッシュボードに通知を表示します。通知内のリンクから、対象の履歴画面を確認できます。

### 6. 月次の勤怠確認
* **実績データの一覧確認**: 店長や人事は、スタッフごとの実働時間、残業時間、遅刻・早退の回数などを月単位で確認できます。
* **月次確定（締め処理）**: 月ごとの勤怠実績を確認して確定します。確定された勤怠実績は、確定解除されるまで打刻変更や修正申請ができません。

* **`dokoTsubu-master/` (アプリケーション本体)**

## ローカル環境での起動手順

ローカル環境でアプリをビルドし、起動するための概要手順です。詳しい設定は [インフラ・デプロイ手順書 (deployment.md)](docs/deployment.md) にまとめています。

### 1. 前提環境
* Java 21以上
* Apache Tomcat 10.1 がローカルにインストールされていること

### 2. ビルドおよび起動
PowerShell環境で、まずアプリケーション本体のフォルダへ移動し、以下のスクリプトを実行します。

```powershell
# アプリケーション本体のフォルダへ移動
cd dokoTsubu-master

# アプリケーションのビルド（Tomcatで動かすための完成ファイルを作成）
powershell -ExecutionPolicy Bypass -File .\build.ps1

# ローカルTomcatの起動（デフォルトで H2 データベースを使用）
powershell -ExecutionPolicy Bypass -File .\run-dev.ps1
```
起動後、`http://localhost:8080/` にアクセスするとシステムを確認できます。

## 工夫した点

1. **役割に応じたアクセス制御**
   * ログイン状態やユーザーの役割を確認し、権限のない画面やデータにはアクセスできないようにしています。これにより、従業員が管理画面を開いたり、店長が原則として自分と同じ拠点・部署以外のデータを操作したりすることを防ぎます。
2. **安定したデータベース接続**
   * クラウド上の PostgreSQL (Supabase) に安定して接続できるよう、**Supabase Session Pooler** を経由した接続方法を採用しています。また、HikariCPを使って接続を効率よく管理し、本番環境の限られたリソースでも安定して動作するようにしています。
3. **エラー画面での情報保護**
   * システムエラーが発生した場合でも、利用者の画面にデータベース接続情報や詳細なエラー内容が表示されないようにしています。本番環境では、必要な情報はログに記録し、画面には安全なエラー表示を行うようにしています。
4. **公開前のセキュリティ確認と改善**
   * ポートフォリオとして安心して確認できるよう、公開前に認証、権限、入力内容、セッション、ログ出力を見直しました。初期管理者の安全確認、ログインやパスワード再設定の連続操作制限、勤怠修正申請の入力チェックを強化し、本番環境では機密情報や詳しいエラー内容を画面・ログに出さないようにしています。
   * 他拠点のシフト表はヘルプ調整のために閲覧できる仕様を維持しながら、社員番号、詳細メモ、未確定シフトなど、調整に不要な情報は表示しないようにしています。
