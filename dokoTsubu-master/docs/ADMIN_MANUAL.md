# ShiftFlow 管理者向け操作マニュアル

## 1. 対象者と権限

| 役割 | 主な範囲 |
|---|---|
| 店長 | 自分の営業所・部署のシフト調整、申請承認、勤怠確認・月次確定、代理者設定 |
| 代理者 | 設定された有効期間中、担当店長と同じ営業所・部署の申請処理 |
| 人事担当者 | 全社データ、従業員・資格・マスター・設定、出力、監査、メール再送 |

権限はログインのたび、また各リクエストで最新状態が確認される。異動・無効化後に古いセッションで権限を維持することはできない。

## 2. ログインとアカウント

1. 組織から案内されたShiftFlowのURLを開く。
2. 登録メールアドレスとパスワードを入力する。
3. 5分以内に5回失敗すると、5分間ログインできない。繰り返し試さず、登録内容を確認する。
4. パスワードを忘れた場合は「パスワードをお忘れですか？」から再設定する。リンクは24時間・1回限り有効である。
5. 右上のアカウント設定で表示言語とパスワードを変更できる。

パスワード、招待・再設定トークンを他者へ伝えない。共有アカウントを作らない。

## 3. ダッシュボード

ダッシュボードには、今日の勤務者、未承認申請、有休残数、人員不足、月間勤務時間が権限範囲内で表示される。勤務・残業・有休取得の月別グラフも確認できる。

- 店長は担当営業所・部署の数値を確認する。
- 人事担当者は全社傾向を確認する。
- 人員不足や未承認件数に異常がある場合は、対象画面で明細を確認する。

## 4. 初期設定（人事担当者）

本番では `shiftapp.seedDemo=false` とし、デモデータを使用しない。次の順序で登録する。

1. 「マスターデータ」→「営業所」で営業所を登録する。
2. 「部署」で部署を登録する。
3. 「雇用形態・資格名称管理」で雇用形態と資格名称を登録する。
4. 「勤務区分・休憩時間管理」でコード、日英名称、開始・終了、休憩時間を設定する。
5. 「必要人数管理」で勤務区分ごとの必要人数を設定する。
6. 法令・運用設定と有休付与ルールを、承認済み規程に合わせて設定する。
7. 初期人事担当者、店長、従業員を登録する。

使用済みマスターは削除せず「無効」にする。勤務区分コードの意味を途中で変更しない。

## 5. 従業員管理（人事担当者）

### 登録と招待

1. 「従業員」→「従業員を登録」を開く。
2. 社員番号、氏名、メール、入社日、営業所、部署、雇用形態、役割を入力する。
3. 「登録して招待」を押す。
4. 招待メールが送信待ちへ登録されたことを「通知」のメール送信状況で確認する。

社員番号とメールアドレスは重複登録できない。登録前に人事台帳と照合する。招待リンクは24時間・1回限り有効である。

### 変更・無効化

従業員一覧の「編集」から所属、役割、メールなどを変更する。退職・利用停止時は削除せず無効化する。無効化後はログインできず、既存セッションも次の操作時に終了する。

所属・役割変更は権限範囲に直結するため、適用日、申請者、承認者を作業記録へ残す。

## 6. 資格情報（人事担当者）

1. 資格名称マスターを先に登録する。
2. 「資格情報管理」で従業員、資格名、有効期限を選択して登録する。
3. 更新時は名称、有効期限、状態を変更する。

過去記録は削除せず無効化し、履歴を残す。資格は情報管理用途であり、必要人数や自動配置条件には使用されない。

## 7. 代理店長

1. 「代理店長設定」を開く。
2. 人事担当者は店長、代理者、開始日、終了日を選択する。店長は自分の代理者を設定する。
3. 代理者は店長と同じ営業所・部署の有効な利用者を指定する。
4. 期間終了・取消時は状態を無効にする。

代理権限は指定期間内だけ有効である。長期間・無期限の代理設定は避け、復帰時に無効化を確認する。

## 8. シフト管理

### 希望提出状況の確認

翌月希望の提出期限は毎月15日、未提出通知は前日の14日に送られる。店長は提出状況と通知状況を確認し、未提出者へ連絡する。

### 調整

1. 「シフト調整」で対象月を選ぶ。
2. 従業員、日付、勤務区分、状態、備考を入力する。
3. 保存後、人員不足、夜勤明け休息などの警告を確認する。
4. 営業所・部署外の従業員は編集できない。

### 確定

1. 「シフト確定」で対象月と警告一覧を確認する。
2. 警告が残る場合は内容を業務上確認し、確定理由を入力する。
3. 「警告を確認して確定」を押す。
4. 従業員への通知とメール送信状況を確認する。

確定理由に個人情報、有休理由、位置情報を書かない。

### 変更・休み申請

申請一覧で変更前後、理由、緊急表示を確認して承認または却下する。承認すると対象日のシフトへ反映される。代理者が処理する場合は代理期間を先に確認する。

## 9. 有休管理

### 承認

1. 「有休承認」を開く。
2. 取得日、単位、時間、理由、現在残数を確認する。
3. 全日、午前、午後、時間単位の整合性を確認して承認または却下する。

承認により付与残から期限順に消費され、取消時は残数が戻る。時間単位の年間上限、出勤率、繰越、失効は承認済み有休ルールに従う。

理由は要配慮情報として扱い、メール・ログ・不要なメモへ転載しない。

### 有休ルール

人事担当者は施行日ごとに出勤率基準、時間単位上限、1日時間、有効月数、取得義務日数を登録する。既存期間へ遡って変更する場合は、就業規則と影響対象を確認してから実施する。

## 10. 勤怠管理

### 日次確認

「勤怠確認・月次確定」で出勤、退勤、遅刻、早退、残業、位置情報取得状態を確認する。位置情報は打刻記録の確認だけに使用し、不正判定や打刻制限には使用しない。

### 打刻修正

1. 修正申請で現在値、申請値、理由を確認する。
2. 対象月が確定済みの場合は、必要な範囲だけ確定解除する。
3. 承認または却下する。
4. 承認後の値と監査ログの変更前後を確認する。

DBを直接修正しない。画面の申請・承認を使用して履歴を残す。

### 月次確定

1. 対象月を選択する。
2. 打刻漏れ、未処理修正、異常な勤務時間がないことを確認する。
3. 従業員別確定、または担当範囲の月次一括確定を行う。
4. 確定後は打刻・修正・承認ができないことを確認する。

修正が必要な場合だけ確定解除し、修正承認後に再確定する。人事担当者は全社の確定状況を確認する。

## 11. 通知とメール（人事担当者）

通知画面のメール送信状況で次を確認する。

| 状態 | 意味 | 対応 |
|---|---|---|
| QUEUED | 送信待ち | 通常は待機する |
| RETRY | 自動再試行待ち | SMTP障害と次回時刻を確認する |
| FAILED | 最大試行回数到達 | SMTP復旧後に「再送」を実行する |
| SENT | 送信完了 | 対応不要 |

失敗メールを別経路で並行送信すると重複するため、復旧後はShiftFlowの再送機能を使う。SMTPパスワードや認証エラー全文を問い合わせへ貼らない。

## 12. データ出力と印刷（人事担当者）

「データ出力」で対象データ、期間、営業所、部署、従業員、CSVまたはExcelを指定する。出力ファイルには個人情報が含まれるため、承認済み端末・保管場所だけで扱う。

月間シフト表はシフト画面の印刷表示を使用する。印刷物の放置、私用クラウドへの保存、不要なメール添付を禁止する。

## 13. 操作履歴（人事担当者）

開始日、終了日、実行者、操作種別、対象者で検索できる。結果には日時、実行者、操作、対象、変更前後が表示される。

監査ログは修正・削除しない。位置情報、有休理由、パスワード、トークンなどが含まれている疑いがある場合は、閲覧範囲を広げずP1/P2としてシステム責任者へ連絡する。

## 14. 月次運用チェックリスト

1. 翌月希望の未提出者通知を確認する。
2. シフト警告を解消または理由付きで確認し、翌月シフトを確定する。
3. 有休・シフト変更・打刻修正の未処理を0件にする。
4. 打刻漏れ、遅刻・早退、残業を確認する。
5. 店長が担当範囲を月次確定する。
6. 人事担当者が全社確定状況を確認する。
7. 必要なCSV・Excel・印刷物を承認済み保管場所へ保存する。
8. メール失敗、日次処理失敗、監査上の異常がないことを確認する。

## 15. 禁止事項

- 稼働中のH2 `.mv.db` を直接コピーしない。
- 本番DBを復元先に指定しない。
- DBを手動更新して申請・承認・監査を迂回しない。
- パスワード、招待・再設定トークン、SMTP秘密情報を共有しない。
- 位置情報や有休理由を不要なメール、チケット、ログへ転載しない。
- デモアカウントを本番で有効にしない。

障害・復旧は [運用ランブック](OPERATIONS_RUNBOOK.md)、移行は [本番移行・切り戻し手順](MIGRATION_ROLLBACK.md)、バックアップは [H2バックアップ・復元検証](../ops/BACKUP.md) に従う。
