# シフト・有休管理アプリ 面接説明用要約

面接やポートフォリオ提出時に、面接官へ本システムの目的、技術構成、工夫点、および開発プロセス（AIとの協働）について 1〜3 分で簡潔にアピールするための説明用要約です。

---

## 1. プレゼンテーション構成案 (想定時間: 2分)

### 【導入】システム概要と開発の背景 (30秒)
* **テーマ**: 複数拠点・複数部署の従業員を抱える組織向けの「シフト・有休・勤怠の一元管理システム」です。
* **背景と機能**: 
  紙やスプレッドシートで行われていた「月次希望シフト提出・調整・確定」「有休の申請と出勤率に応じた自動付与・残管理」「打刻（GPS位置情報の記録）」「代理店長の設定と不在時の承認代行」「操作履歴（監査ログ）」などを、役割（人事担当者・店長・従業員）ごとの認可制御に沿って、Webブラウザ（PC・スマホ対応）から一元管理できるようにしました。

### 【技術的ハイライト】開発の工夫点 (1分)
* **ポイント1: SQL動的互換レイヤーの設計**
  「開発時は手軽なファイル型（H2）」を使い、「本番は高信頼なマネージドDB（Supabase PostgreSQL）」を使うマルチDBアーキテクチャを実装しました。H2とPostgreSQLの日時演算・集計関数の差異（`DATEADD` 対 `INTERVAL` 等）を、Javaサービス層の `Database.isPostgres()` 判定により動的にクエリ置換・分岐させるレイヤーを自作しました。これによりコードベースを1本に維持したまま両立させました。
* **ポイント2: 徹底的なセキュリティガード（安全装置）**
  ポートフォリオとして一般公開するため、秘密情報はすべて環境変数から注入するように設計しました。さらに、初期管理者アカウントや自動生成デモアカウントのパスワードが初期ダミー値（`Password1!`）のまま本番稼動した場合には、安全装置として起動時例外を投げて Tomcat 自体を停止させる「防衛的プログラミング」を徹底しました。また、ログイン画面からの情報公開を廃止し、個別に面接時に共有する運用としています。
* **ポイント3: Docker化とRenderコンテナ運用への適合**
  Render（Web Service）のコンテナ環境に合わせ、Tomcatのポート設定 (10000) や不要なシャットダウンポート (8005) の無効化、および Render Free Plan の仕様に伴う初回起動スリープ復帰の注意書きをドキュメントに配置するなど、実際の公開・運用フェーズを見据えたインフラ設計を行いました。

### 【開発プロセス】生成AIを活用した主体的なアジャイル開発 (30秒)
* **AIとの関わり方**: 
  本システムは生成AIを開発パートナーとして活用しながら、自身が主体となって要件整理・実装・検証・エラー原因の切り分けを行いました。
  * AIが生成したコードをそのまま使うのではなく、Render LogsやSupabaseのDB状態を確認しながら自ら動作検証とバグ修正を繰り返しました。
  * H2からPostgreSQL移行時に発生したデータ型やSQL関数のエラーは、ログから問題箇所を自ら特定し、SQL互換性対応や型バインドの修正を設計・実装しました。
  * リファクタリングについても、保守性の向上・責務分離・重複削減を目的として、自身のコントロール下で実施しました。
  * AIを単なる「自動生成ツール」として丸投げするのではなく、アイデア出しやトラブルシューティングの壁打ち役として高度に協働し、短期間で品質の高いWebシステムを構築しました。

## 2. 想定質問と回答集 (Q&A)

#### Q1. なぜ Spring Boot などのフレームワークではなく、生の Java Servlet/JSP を選んだのですか？
* **回答**: 
  フレームワークの裏側で自動的に行われている「HTTPリクエストのハンドリング」「セッション管理」「Servletライフサイクル」「サーブレットフィルタによる文字エンコードやセッション認可」といった、**Webアプリケーションの最も基礎的な仕組みを深く理解したかったため**です。生の Servlet を手動で記述し、自前で DAO やサービス層を設計・結合したことで、Java Web 技術のコア部分が非常にクリアに理解できました。

#### Q2. 開発環境（H2）と本番（PostgreSQL）で SQL 互換性の対応をしたとのことですが、最初から PostgreSQL 一本で開発しなかったのはなぜですか？
* **回答**: 
  **オフラインでも動作し、コマンド一つでミリ秒単位でDB付きのテストを完全初期化して実行できる高速な開発環境**を維持したかったためです。ローカルH2データベースにより、ネットワーク接続に依存せず `test.ps1` を使って数十件の結合・E2Eテストを数分で一括実行できるため、開発のイテレーションが圧倒的に速くなりました。その上で、本番移行の課題を動的互換レイヤーで克服するという技術的アプローチ自体が、良い設計の訓練になりました。

#### Q3. 安全装置（例外を投げて起動停止）はなぜ必要なのですか？
* **回答**: 
  GitHub の公開リポジトリや Render のデモ公開サーバーが、**「デフォルトのパスワードのまま放置される」ことによる不正アクセス・スパムデータのインサートを防ぐため**です。人事マスタや有休、監査ログなどを扱うシステムであるため、本番環境起動時に環境変数で強力なカスタムパスワードが設定されているかを強制チェックする安全機構は、システム全体の保全のために必須であると考え設計しました。
