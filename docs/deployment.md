# インフラ・デプロイ手順書

このドキュメントでは、シフト・勤怠管理アプリを自分のパソコンで動かす方法と、想定本番構成である Render / Supabase を使って公開するための設定を説明します。現在の稼働状態や実際の設定値はリポジトリだけでは確認できないため、デプロイ時に確認日とあわせて確認してください。

---

## 1. インフラ構成の概要

このアプリは、Webアプリを動かすための環境と、データを保存するためのデータベースを組み合わせて動いています。

* **Webアプリを動かす環境**: **Render Web Service**
  * Dockerコンテナを使って、Tomcat 10.1 と Java 21 が動く環境を用意しています。
  * ビルドで作成した `ROOT.war` を Tomcat に配置し、アプリのトップページとして表示されるようにしています。
* **データベース環境**: **Supabase PostgreSQL**
  * クラウド上の PostgreSQL データベースを使い、従業員・シフト・勤怠・申請などのデータをまとめて管理しています。

---

## 2. Supabaseへの接続と Session Pooler の採用

本アプリは、Render 上で動くアプリから Supabase PostgreSQL へ接続する際に、**Supabase Session Pooler** を利用する構成を前提として設定・検証しています。

### Session Pooler を使う理由

1. **データベース接続を安定させるため**
   * Session Pooler を経由することで、データベースへの接続数を適切に制御し、接続の確立・切断による負荷を軽減しています。これにより、接続上限に達しにくくなり、安定した接続を維持しやすくなります。

2. **接続数を効率よく管理するため**
   * 本アプリでは、アプリケーション内の接続プールを HikariCP、データベースとの接続管理を Session Pooler が担当しています。二段階で接続を管理することで、限られた接続数でも安定した動作を実現しています。

> 注意: 本アプリは Session Pooler を前提として設定・検証を行っています。Transaction Pooler など他のプーリングモードを使用する場合は、接続設定やアプリケーションの動作を別途確認してください。

## 4. Docker / Tomcatの役割

### 4.1 Dockerfile によるアプリの準備

このプロジェクトでは、`Dockerfile` を使って、アプリを動かすための環境を自動で準備しています。

1. **ビルドする段階**
   * Java のソースコードをコンパイルし、Tomcat で動かせる `ROOT.war` という完成ファイルを作成します。
   * このとき、本番環境で PostgreSQL に接続するために必要な JDBC ドライバーも自動で含めます。

2. **アプリを実行する段階**
   * Tomcat 10.1 と JDK 21 が入った公式イメージを使い、作成した `ROOT.war` を Tomcat に配置します。
   * これにより、ローカルの Docker 環境でも Render 上でも、できるだけ同じ状態でアプリを動かせるようにしています。

## 5. ローカル環境での起動手順

ローカル環境でアプリを確認する方法は、主に2つあります。

### 5.1 PowerShell スクリプトでローカル Tomcat を起動する方法

自分のパソコンに Java 21 と Tomcat 10.1 が入っている場合は、以下のスクリプトでビルドと起動を行えます。

```powershell
# 1. アプリをビルドする
powershell -ExecutionPolicy Bypass -File .\build.ps1

# 2. ローカルTomcatでアプリを起動する（通常は H2 データベースを使用）
powershell -ExecutionPolicy Bypass -File .\run-dev.ps1
```

起動が完了したら、ブラウザで `http://localhost:8080/` にアクセスします。

### 5.2 ローカル Docker で起動する方法

Tomcat を自分のパソコンに直接用意せず、Docker コンテナとしてアプリを動かす方法です。

```powershell
# 1. コンテナイメージを作成する
docker build -t shiftflow .

# 2. ポート10000でコンテナを起動する
docker run --rm -p 10000:10000 shiftflow
```

起動が完了したら、ブラウザで `http://localhost:10000/` にアクセスします。

## 6. 本番環境での注意事項とトラブルシューティング

### ログイン状態を守るCookieの確認

本番環境では、ログイン状態を安全に保つため、セッションCookie（ブラウザがログイン状態を覚えるための情報）を保護しています。Renderへ公開した後、ブラウザの開発者ツールで、Cookieに `HttpOnly`、`Secure`、`SameSite=Lax`、`Path=/` が設定されていることを確認してください。これらは、Cookieの盗み見や他サイトからの不正利用を起こりにくくする設定です。

パソコン上のローカル環境はHTTPで動かすため、`Secure` が表示されない場合があります。本番のRender環境はHTTPSであるため、`Secure` が表示されていることが必要です。

### 6.0 ヘルスチェックと起動時間の確認

Render の Health Check Path には `/health` を指定します。このエンドポイントはデータベースへ接続せず、アプリケーションの起動完了後に `200 OK` と `ok` を返します。

起動時には Render のログへ `Startup timing:` で始まる計測結果が出力されます。`database.connection`、`database.schema`、`database.base-seed`、`database.total`、`application.bootstrap` を比較することで、接続・スキーマ確認・初期データ確認・起動全体のどこに時間がかかっているか判別できます。

### 6.1 Render Freeプランによる起動遅延について

Render の無料プランを使用する構成では、一定時間アクセスがない場合、アプリが自動的に休止状態になることがあります。

休止後に最初にアクセスしたときは、アプリケーション（Tomcat）の起動やデータベースへの接続確認が行われるため、通常より時間がかかることがあります。
画面が表示されるまでしばらく待ってから操作してください。
