## 3. 開発過程における工夫点 (面接アピール用)

### ① マルチDBに対応する「SQL動的互換レイヤー」の自作
開発時の生産性を高めるための軽量な「H2」と、本番環境の「PostgreSQL」を共存させるため、動的に SQL 構文を書き換える仕組みを実装しました。
* `Database.isPostgres()` メソッドを実装し、実行中のデータベースが PostgreSQL かどうかを判定。
* PostgreSQL 時には `DATEADD('day', 1, work_date)` を `(work_date + INTERVAL '1 day')::date` に書き換えるなど、SQL 内の独自関数や日時計算の差異を Java サービス層で安全に置換・分岐するレイヤーを構築しました。
* これにより、コードベースを一本化し、ローカルでの動作確認の容易性と本番環境の堅牢性の両立に成功しました。

### ② 本番環境のセキュリティ安全装置（デフェンシブプログラミング）
ポートフォリオとして一般公開した際、悪意ある第三者による不正アクセスや情報漏洩を完全に防ぐための安全装置を Java 起動初期化処理に組み込みました。
* 環境変数 `DEMO_ACCOUNTS_ENABLED=true` の場合のみデモアカウントが自動生成されます。
* 初期 HR 管理者 (`hr@example.com`) やデモアカウントのパスワードが、デフォルトの `Password1!`（ダミー値）のままで本番デプロイされた場合、アプリは `IllegalStateException` を発生させて起動自体を強制停止します。
* ログイン画面や公開リポジトリの README から、具体的なアカウント情報を一切排除し、パスワードを暗号ハッシュ化（`PasswordUtil.hash`）して保存するセキュリティ水準を担保しました。

### ③ 生成AIを活用したアジャイルペアプログラミング
設計・実装・テスト・インフラ構築のすべての過程において、生成AIを活用しました。
* AI を優秀なシニアエンジニアとして扱い、H2 と PostgreSQL の型変換エラー（TIMEバインド、CLOBの差異）の解決法や、Docker コンテナの Tomcat シャットダウンポート衝突といった複雑なインフラエラーのトラブルシューティングを高速に実施。
* 「AI に仕様を丸投げする」のではなく、「自身でアーキテクチャの方針をコントロールしつつ、具体的な実装コードやテスト自動化スクリプトの作成、エラーログ分析をAIに分担させる」ことで、開発効率を約3倍に高めました。

## 4. 苦労した点と解決方法

### 課題1: H2 と PostgreSQL での関数・集計仕様の差異による SQL エラー
* **状況**: 開発環境の H2 では動いていた有給計算やシフト集計処理が、本番 PostgreSQL 移行時に `DATEADD` や `FORMATDATETIME` の未定義エラーとなり動作しなくなりました。
* **解決方法**: 
  `Database.java` に `isPostgres()` 判定処理を追加し、`DashboardService` や `ShiftService` でクエリを投げる際に `Database.isPostgres()` の真偽値に基づいて SQL 文字列を置換・分岐させるロジックを実装。また、TIME 型バインドにおける JDBC の `setString` から `setTime` への変更等を行い、互換性を担保しました。

### 課題2: Render Free Plan 上での Tomcat の動作・ヘルスチェック失敗
* **状況**: Render のヘルスチェック時に Tomcat のシャットダウンポート (8005) のアクセスが拒否される、あるいはスリープ復帰時 (Free Plan) にポートバインドが競合してコンテナがクラッシュを繰り返す問題が発生しました。
* **解決方法**: 
  `Dockerfile` のビルドスクリプト内で、Tomcat の `server.xml` を書き換えてシャットダウンポートを完全に無効化（`-1` に設定）し、待ち受けポートを Render の期待する `10000` に合わせる構成を実装。コンテナ挙動を最適化することで、安定起動を実現しました。
