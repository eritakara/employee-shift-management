# システム設計書

このドキュメントでは、シフト・勤怠管理アプリがどのような仕組みで動いているかを説明します。

アプリ全体の構成、画面が表示される流れ、フォルダの役割、利用者の権限ごとにできること、主な処理の流れをまとめています。

---

## 1. システム構成

このアプリは、Webアプリを動かすための環境と、データを保存するためのクラウドデータベースを組み合わせて動いています。

想定本番構成では、利用者がPCやスマートフォンからアクセスすると、Render上で動くTomcatがリクエストを受け取り、必要な処理を行ったうえで、Supabase PostgreSQLにデータを保存・取得します。現在の稼働状態や実際の接続先は、共有時に別途確認が必要です。

```mermaid
graph TD
    User([ユーザー: PC / スマホ]) -->|HTTPS| WebServer[Render: Tomcat 10.1 Docker Container]
    subgraph WebServer
        Filter[AuthenticationFilter: セッション・ログイン検証] --> Servlet[PortalServlet: 画面ディスパッチ・制御]
        Servlet --> Service[Service層: 業務ロジック処理]
        Service --> DAO[DAO / Sqlクラス: データベース接続]
    end
    DAO -->|JDBC SSL経由| DB[(Supabase PostgreSQL: Session Poolerポート経由)]
```

* **ホスティング**: Render Web Service
  * Dockerコンテナを使い、Tomcat 10.1 と Java 21 が動く環境を用意しています。これにより、アプリを本番環境でも同じ形で動かしやすくしています。
* **データベース**: Supabase PostgreSQL
  * 勤怠情報、シフト情報、ユーザー情報などを保存します。想定本番構成では、**Supabase Session Pooler** を前提に接続設定と検証を行っています。

## 2. アプリケーション構成（役割分担）

このアプリは、画面を表示する部分、利用者の操作を受け取る部分、実際の処理を行う部分、データベースとやり取りする部分を分けて作っています。

### 画面に表示する部分

利用者が見る画面を作る部分です。

主に `WEB-INF/jsp/app.jsp`、CSS、JavaScript を使って、ログイン後の画面やシフト確認画面、勤怠打刻画面などを表示します。

### 操作を受け取る部分

利用者がボタンを押したり、フォームを送信したりしたときに、その操作を受け取る部分です。

たとえば、`PortalServlet.java` や `AuthServlet.java` が該当します。
どの画面を表示するか、どの処理を呼び出すかを判断します。

### 事前チェックを行う部分

画面を表示する前に、ログインしているか、アクセスしてよい画面かを確認する部分です。

たとえば、`AuthenticationFilter.java` が該当します。
未ログインのまま管理画面へ進めないようにしたり、権限のない画面へアクセスできないようにしたりします。

### 実際の業務処理を行う部分

打刻、シフト作成、申請の承認・却下など、アプリの中心となる処理を行う部分です。

たとえば、`AttendanceService.java` や `ShiftService.java` が該当します。
画面から受け取った内容をもとに、必要な確認を行い、勤怠やシフトの情報を更新します。

### データベースとやり取りする部分

従業員情報、シフト情報、勤怠実績などをデータベースから取得したり、登録・更新したりする部分です。

たとえば、`Sql.java` が該当します。
アプリの処理結果をデータとして保存したり、画面に表示するための情報を取り出したりします。

## 3. ディレクトリ構成

アプリ本体は、以下のようなフォルダ構成になっています。

それぞれのフォルダごとに役割を分けることで、どこに何の処理があるかを確認しやすくしています。

```text
dokoTsubu-master/
├── src/
│   ├── main/

│   │   │   ├── model/         # ユーザーなどのデータを表すクラス

│   │       └── WEB-INF/
```

## 4. 権限ごとの利用可能機能

このアプリでは、ログインする人の役割によって、利用できる機能が変わります。

たとえば、一般従業員は自分の勤怠やシフトを確認できます。店長は所属スタッフの勤怠やシフトを管理できます。人事は従業員アカウントやマスタデータの管理も行えます。

`PortalServlet.java` の `allowed` メソッドなどで、権限に応じたアクセス制御を行っています。

| 機能（画面パス） | 内容 | 従業員 (EMPLOYEE) | 店長 (MANAGER) | 人事 (HR) |
| :--- | :--- | :---: | :---: | :---: |
| `dashboard` | 概要・通知表示・月間シフト | ◯ | ◯ | ◯ |
| `attendance/clock` | 出退勤の打刻 | ◯ | ◯ | ◯ |
| `attendance/mine` | 自分の勤怠実績確認 | ◯ | ◯ | ◯ |
| `attendance/adjust` | 勤怠修正の申請作成 | ◯ | ◯ | ◯ |
| `attendance/history` | 自分の申請履歴 | ◯ | ◯ | ◯ |
| `shifts/mine` | ヘルプ調整用の拠点別確定済み月間シフト表（閲覧専用） | ◯ | ◯ | ◯ |
| `attendance/manage` | 所属スタッフの月次勤怠確認・確定 | ✕ | **◯** | **◯** |
| `shifts/manage` | カレンダーでのシフト作成・調整 | ✕ | **◯** | **◯** |
| `employees` | 従業員アカウント管理 | ✕ | ✕ | **◯** |
| `masters/*` | 店舗・部署・勤務区分などの基本データ管理 | ✕ | ✕ | **◯** |

`shifts/mine`では、欠勤時などの拠点間ヘルプを調整しやすくするため、一般従業員と店長も他拠点を選んで確定済み月間シフト表を閲覧できます。公開する項目は氏名、勤務日、勤務区分などに限定し、社員番号や詳細メモは含めません。この閲覧経路には編集・削除・承認機能を設けず、店長の管理操作は原則として自分と同じ拠点・部署、人事の管理操作は全体という既存の権限制御を維持します。

## 5. 主な処理の流れ

ここでは、利用者の操作に対して、アプリの中でどのように処理が進むかを説明します。

### 5.1 出退勤打刻処理

従業員がポータル画面で「出勤」または「退勤」ボタンを押したときの流れです。

1. **画面操作の受け取り**
   * `PortalServlet.doPost` が、出勤または退勤の操作を受け取ります。
2. **勤怠処理の実行**
   * `AttendanceService.clock` を呼び出し、現在時刻を取得します。
   * すでに打刻済みではないかを確認します。
3. **データベースへの登録・更新**
   * その日の勤怠データがまだない場合は、新しく作成して出勤時刻を記録します。
   * 退勤の場合は、既にある勤怠データに退勤時刻を記録します。
   * 確定シフトと実際の打刻時刻を比べて、遅刻や早退があるかを自動で判定します。
4. **画面の更新**
   * 打刻後は元の打刻画面に戻り、最新の打刻時間が表示されます。

### 5.2 勤怠修正申請の承認・却下処理

承認権限を持つ担当者が、届いた勤怠修正申請を確認し、承認または却下する流れです。

1. **申請への操作を受け取る**
   * `PortalServlet.doPost` が、承認または却下の操作を受け取ります。
2. **承認・却下の判定処理を行う**
   * `AttendanceService.decideAttendanceAdjustment` を呼び出し、処理を実行します。
   * 操作している人が、店長、人事、または有効な代理店長として承認権限を持つことを確認します。
   * 自己承認はできません。
   * 店長の場合は、原則として申請者が自分と同じ拠点・部署に所属しているかを確認し、担当範囲外のデータを操作できないようにします。
   * 人事は権限規則に従って処理します。また、設定された代理店長が処理できる場合もあります。
   * すでに承認または却下済みの申請ではないかを確認します。
3. **データベースを更新する**
   * 承認の場合は、申請された時間に合わせて勤怠実績を更新します。
   * 却下の場合は、勤怠実績は変更せず、申請の状態だけを「却下」に変更します。
4. **申請者へ通知する**
   * 承認または却下の結果を、申請した従業員へシステム内通知として送ります。
5. **管理画面へ戻る**
   * 処理が終わると、管理用の一覧画面に戻ります。

## 6. 設計上工夫した点

* **画面構成をまとめて管理**
  * 各画面の表示を `app.jsp` にまとめ、メニューやヘッダーを共通化しています。これにより、画面ごとの見た目をそろえやすくし、権限に応じた表示切り替えも管理しやすくしています。
* **データベース操作を共通化**
  * `Sql.java` にデータベース接続やSQL実行の処理をまとめています。これにより、接続漏れを防ぎやすくし、SQLインジェクション対策も行いやすくしています。
* **画面操作と業務処理を分けて管理**
  * Servletは画面からの操作を受け取り、Serviceクラスは勤怠・シフト・通知などの処理を担当します。役割を分けることで、修正や確認がしやすい構成にしています。
