# ShiftFlow 招待メール一斉送信手順

## 1. 目的と安全原則

初期利用者を登録し、招待メールを安全に送信・照合する。パスワードや招待トークンをCSV、チケット、チャットへ記載しない。招待完了前のアカウントには推測不能なランダム値が設定されており、共通初期パスワードではログインできない。

招待リンクは24時間・1回限り有効である。再発行すると以前の未使用リンクは直ちに無効になる。

## 2. 役割

| 役割 | 責任 |
|---|---|
| 人事担当者 | 対象者、社員番号、氏名、メール、所属、役割の確定と登録 |
| システム担当者 | 本番URL、SMTP、送信元、TLS、メール送信処理の確認 |
| 確認者 | 登録件数、送信結果、重複・誤送信の相互確認 |
| 業務責任者 | 送信日時、案内文面、利用開始日の承認 |

## 3. 送信前チェック

1. 本番URL、DNS、TLS証明書が正しく、外部から招待URLへ接続できることを確認する。
2. `SHIFTFLOW_SMTP_*` と送信元を秘密管理先から設定し、秘密値を作業票へ転記しない。
3. 営業所、部署、雇用形態、役割のマスターが確定していることを確認する。
4. 対象者一覧を二者確認し、社員番号とメールアドレスの重複、退職者、送信対象外を除外する。
5. 送信予定人数、送信日時、問い合わせ先、再送担当者を記録する。
6. 少人数の承認済みテスト対象で、件名、本文、URL、24時間期限、パスワード設定、ログインを確認する。
7. メール送信状況に過去の `FAILED` や大量の `RETRY` がないことを確認する。

## 4. 登録と一斉送信

1. 人事担当者でログインし、「従業員」→「従業員を登録」を開く。
2. 社員番号、氏名、メール、入社日、営業所、部署、雇用形態、役割を確定一覧と照合して入力する。
3. 「登録して招待」を押す。登録と同時に招待メールが送信待ちへ入る。
4. 対象者ごとに登録結果を一覧へ記録する。画面エラーが出た対象は原因を確認し、同じ情報を連続登録しない。
5. 初期データ移行を使う場合も、登録件数と送信待ち件数を照合してからメール送信処理を有効にする。
6. メール送信処理は60秒ごとに最大50件を取得する。100件の場合、正常時は複数回に分けて処理されるため、途中で手動再送しない。

## 5. 送信結果の照合

人事担当者が「通知」のメール送信状況を確認する。

| 状態 | 対応 |
|---|---|
| QUEUED | 送信処理を待つ。件数が継続して減ることを確認する |
| RETRY | 1分、5分、15分の自動再試行を待ち、SMTP状態を確認する |
| FAILED | SMTP復旧後、対象メールの「再送」を1回実行する |
| SENT | 送信完了として対象者一覧へ記録する |

次の件数を照合し、差異が0件になるまで利用開始案内を出さない。

- 登録承認済み人数
- ShiftFlowの新規有効アカウント数
- 招待メール作成数
- `SENT`、未送信、失敗の合計

メール本文や招待URLを作業票へコピーしない。宛先を記録する場合は、承認済みのアクセス制限された場所を使用する。

## 6. 未着・期限切れ・誤登録

### メール未着

1. 対象者に迷惑メールと登録メールアドレスを確認してもらう。
2. メール送信状況が `FAILED` なら、SMTP復旧後にメール行の「再送」を使用する。この操作では同じ有効リンクが再送される。
3. `SENT` でも届かない場合は、メールサーバーの配送記録をシステム担当者が確認する。

### 招待リンクの期限切れ・紛失

1. 本人の社員番号、氏名、登録メールを承認済み方法で照合する。
2. 人事担当者が従業員一覧の「招待再発行」を押す。
3. 新しいリンクが作成され、以前の未使用リンクが無効になったことを監査記録で確認する。
4. 新しいメールの送信状態を確認する。

メール障害だけの場合に「招待再発行」を使うと、利用者が後から開いた古いメールが無効になる。まずメール行の再送を使用する。

### メールアドレス・所属の誤り

1. 誤った招待を再送しない。
2. 従業員編集で承認済みの正しい情報へ修正する。
3. 情報漏えいの可能性がある場合は運用ランブックに従い、個人情報管理責任者へ連絡する。
4. 正しいメールアドレスへ招待を再発行する。

## 7. 利用開始確認

1. 対象者が招待リンクで10文字以上のパスワードを設定する。
2. 本人がログインし、自分の社員番号、メール、所属を確認する。
3. 誤りがある場合は操作を続けず、人事担当者へ連絡する。
4. 人事担当者は未使用・期限切れ招待の残数、送信失敗、問い合わせ件数を確認する。
5. 業務責任者へ送信数、設定完了数、未完了数、問題を報告する。

## 8. 証跡

次を保存する。秘密値、パスワード、トークン、メール本文中のURLは保存しない。

- 承認済み対象者一覧の版と承認者
- 登録・送信開始終了日時、作業者、確認者
- 状態別件数と差異確認結果
- 再送・再発行の対象者、理由、実施者、監査ログID
- 誤送信、未着、期限切れ、不具合の対応記録
- 利用開始の承認記録
