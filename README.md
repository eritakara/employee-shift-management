# 従業員シフト・勤怠管理システム

本システムは、スタッフの勤務予定（シフト）の作成・調整と、毎日の就業記録（勤怠）の打刻・確認をWeb上で一元管理できるアプリケーションです。
実務における店舗や事業所の運用フローを想定し、ロール（権限）ごとのアクセス制御やデータ整合性に配慮して構築しています。

## 主な機能

ログインするアカウントの権限（人事、店長、一般従業員）に応じて、表示画面や操作可能なアクションが自動的に切り替わります。

### 1. 権限管理・ログイン機能
* **セキュアなログイン/ログアウト**: 各ロールのアカウントでログインし、それぞれの専用ポータルへ遷移します。ログアウトするとセッションが確実に破棄されます。
* **URL直打ち防止（アクセス制御）**: 従業員が管理画面へ直接アクセスしようとした場合や、店長が別店舗のデータを操作しようとした場合に、サーバー側でアクセスを検知して制御（403 Forbidden）します。

### 2. 出退勤打刻機能
* **Web打刻機能**: 従業員が勤務開始時および終了時にワンクリックで現在時刻を打刻します。
* **自動遅刻・早退判定**: 確定したシフト（予定）と実際の打刻時刻を照合し、遅刻や早退を自動的に算出・記録します。

### 3. シフト作成・調整機能
* **シフト作成・編集**: 店舗責任者（店長）および人事が、カレンダーUIや専用画面を用いて、所属スタッフのシフトを作成・調整します。
* **シフト確認**: 一般従業員は、自身に確定した月間シフトをいつでも一覧で確認できます。

### 4. 勤怠修正申請の作成と承認・却下フロー
* **修正申請の作成**: 従業員が過去の打刻漏れに対して、修正希望時間と理由を入力して申請を作成できます。
* **承認・却下機能**: 店長（または人事）が申請を確認し、承認または却下を判断します。承認されると実績データが即座に更新され、却下の場合は元のデータが維持されます。

### 5. 通知機能
* **申請と処理の通知**: 従業員からの修正申請が届いた際や、店長が申請を承認・却下した際に、システム内のダッシュボードにお知らせ通知が表示されます。通知内の詳細リンクから対象の履歴画面へ直接遷移できます。

### 6. 月次の勤怠確認
* **実績データの一覧確認**: 店長および人事は、所属スタッフの実働時間、残業時間、遅刻・早退の回数などを月単位で確認・集計できます。
* **月次確定（締め処理）**: 月次の勤怠を確定し、確定後は従業員からの新規の修正申請をロックする制御が働きます。

## リポジトリの構成

本リポジトリは以下のディレクトリ構成になっており、アプリケーション本体および関連ドキュメントが配置されています。

* **`dokoTsubu-master/` (アプリケーションのソースルート)**
  * `dokoTsubu-master/src/` 配下に、Java Servlet / JSP などのアプリケーション本体のプログラムやWebアセットが格納されています。
  * ルートの `Dockerfile` は、この `dokoTsubu-master/src/` ディレクトリをビルド対象として参照し、コンテナイメージを構築する設計になっています。
* **`docs/` (ポートフォリオ確認用ドキュメント)**
  * ポートフォリオの確認をスムーズに行えるよう、要件定義・システム設計・データベース・デプロイ手順・テスト仕様などの関連ドキュメントが格納されています。
* **`README.md` / `Dockerfile`**
  * リポジトリのルートに配置されており、本番環境への配備および起動設定を司ります。

## ローカル環境での起動手順

ローカル環境でビルドおよび実行を行うための概要手順です。詳細な設定は [インフラ・デプロイ手順書 (deployment.md)](docs/deployment.md) を参照してください。

### 1. 前提環境
* Java 21以上
* Apache Tomcat 10.1 がローカルにインストールされていること

### 2. ビルドおよび起動
PowerShell環境でリポジトリのルートディレクトリにて、以下のスクリプトを実行します。

```powershell
# アプリケーションのビルド (target/ROOT.war が生成されます)
powershell -ExecutionPolicy Bypass -File .\build.ps1

# ローカルTomcatの起動 (デフォルトで H2 データベースが使用されます)
powershell -ExecutionPolicy Bypass -File .\run-dev.ps1
```
起動完了後、`http://localhost:8080/` にてシステムにアクセスできます。

## 工夫した点

1. **ロール（権限）によるアクセス制御の徹底**
   * コントローラーおよびフィルター層において、ログイン状態や各URLパスに対する権限チェックを徹底しています。従業員が管理画面へ不正にアクセスしようとした場合などは、サーバー側で安全に403 Forbiddenへ遷移させる堅牢な設計を行っています。
2. **接続効率を考慮したデータベース接続設計**
   * クラウド上の PostgreSQL (Supabase) に対する接続効率化と到達性確保のため、**Supabase Session Pooler** を経由したJDBC接続を採用しています。また、HikariCPによるDBコネクションプールを導入し、限られた本番リソース環境下での接続数制限と効率化を両立させています。
3. **エラー画面の情報保護**
   * 万が一のシステムエラー時にも、エラー画面にDB接続文字列やプログラムの例外スタックトレースを表示しないよう、本番環境 (`APP_ENV=production`) に応じたログ出力制御を組み込んでいます。
