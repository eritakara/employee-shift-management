# リリース前チェックリスト

本ドキュメントは、本番環境のWebアプリURLおよびGitHubリポジトリを共有する前に、動作、セキュリティ、個人情報、および機密情報の漏洩がないかを検証するためのチェックリストです。実際のシステム構造と照らし合わせながら整理しています。

---

## 1. 本番環境の確認
本番環境（Render / Supabase）が正しく動作し、アクセス可能な状態であることを確認します。

> [!NOTE]
> **Render Freeプランに関する注意**
> Render Freeプランを使用しているため、しばらくアクセスがない場合、初回アクセス時に起動まで時間がかかることがあります。

- [ ] **Renderの最新デプロイ成功**: Renderのダッシュボードで、最新のデプロイが「Deployed」として成功しているか。
- [ ] **デプロイ対象ブランチの整合性**: デプロイ対象となっているブランチ（通常は`main`）が、GitHub上の最新コードと一致しているか。
- [ ] **シークレットウィンドウでのアクセス**: ブラウザのシークレットウィンドウ（プライベートブラウズ）を使い、キャッシュや既存のCookieに影響されない状態で本番環境URLを開くことができるか。
- [ ] **ログイン画面の初期表示**: 認証されていない状態でアクセスした際、正しくログイン画面が表示されるか。
- [ ] **デモユーザーでのログイン**: 事前に登録されているデモユーザー（人事、店長、一般従業員）のアカウント情報を用いて、正しくログインできるか。
- [ ] **主要画面の500エラー防止**: ログイン後の各ダッシュボード、シフト管理、勤怠の主要画面を遷移した際、サーバーエラー（500エラー）が発生せず、正常に画面が表示されるか。
- [ ] **デモデータの視認性**: 画面上のカレンダーや一覧に、確認しやすい十分な量のデモデータが揃っているか。

---

## 2. 主要機能の確認
アプリケーション（Java Servlet / JSP / Tomcat / PostgreSQL / Supabase）における各ロールの主要な業務フローが、意図通りに動作することを確認します。

### ロール別ログイン・画面表示
- [ ] **人事（HR）ユーザー**: 人事権限アカウントでログインし、全社シフト・従業員管理・マスタ管理などが表示されるか。
- [ ] **店舗責任者（店長/MANAGER）ユーザー**: 店長権限アカウントでログインし、自店舗のシフト作成や各種申請の承認画面が表示されるか。
- [ ] **一般従業員（EMPLOYEE）ユーザー**: 一般従業員権限アカウントでログインし、自身のシフト確認や勤怠打刻画面が表示されるか。

### 勤怠・打刻フロー
- [ ] **出退勤の打刻**: 従業員ユーザーが「出勤」「退勤」ボタンをクリックして現在時刻を打刻でき、打刻結果が勤怠実績に反映されるか。
- [ ] **勤怠修正申請の作成**: 従業員ユーザーが「打刻修正」画面から、過去の打刻漏れ等に対する修正申請（申請時間・理由）を作成できるか。
- [ ] **修正申請の承認処理**: 店長または人事ユーザーが「打刻修正申請」一覧から、申請を「承認」できるか。
- [ ] **修正申請の却下処理**: 店長または人事ユーザーが「打刻修正申請」一覧から、申請を「却下」できるか（却下理由の登録を含む）。
- [ ] **承認後のデータ反映**: 承認完了後、対象の勤怠データが申請された時間に変更されているか。
- [ ] **却下後のデータ不変**: 却下された場合、対象の勤怠データは変更されないまま維持されるか。

### シフト・通知・勤怠管理
- [ ] **シフト管理画面の表示**: カレンダーUIを用いたシフト調整画面が正常に表示されるか。
- [ ] **通知画面の表示**: ホーム画面および通知一覧画面に新着情報が正しく表示されるか。
- [ ] **通知リンクからの遷移**: 通知に表示されているリンクをクリックした際、対象の申請画面や管理画面へ直接遷移できるか。
- [ ] **月次勤怠データの確認**: 従業員の「勤怠実績」や店長の「月次確定」画面から、月ごとの勤怠実績データが一覧で確認できるか。
- [ ] **権限による画面制御**: ログイン中のロールに応じて、サイドバーメニューやアクションボタンが適切に制御されているか。

## 3. セキュリティ確認
Webアプリケーションとして想定されるアクセス制御および情報保護が機能しているかを確認します。

- [ ] **未ログイン時のアクセス制限**: 未ログイン状態で、保護された画面や機能のURL（`/app/*`）へ直接アクセスしようとした際、ログイン画面にリダイレクトされるか。
- [ ] **人事専用画面のアクセス制限**: 一般従業員や店長ユーザーが、人事専用の画面URL（従業員管理やマスタ管理など）へ直接アクセスした際、エラー画面（403 Forbidden）が表示され、アクセスが制限されるか。
- [ ] **店長専用画面のアクセス制限**: 一般従業員ユーザーが、店長専用の画面URL（シフト調整や勤怠管理など）へ直接アクセスした際、同様にアクセスが制限されるか。
- [ ] **管理対象外データの操作制限**: 店長ユーザーが、自身の管理店舗・部署以外の従業員の申請（打刻修正申請など）を操作できないようになっているか。
- [ ] **他人のデータ閲覧・操作制限**: 一般従業員ユーザーがURLのパラメータ（IDなど）を変更して、他人の勤怠情報や申請履歴を閲覧・操作できないようになっているか。
- [ ] **操作ボタンの表示制御**: 勤怠修正の「承認」「却下」ボタンが、承認権限を持つユーザー（店長・人事など）にのみ表示されるようになっているか。
- [ ] **二重操作の防止**: すでに「承認済み」または「却下済み」となった申請に対して、再度承認・却下のアクションが実行できないようになっているか（未処理の申請のみを受け付ける制御）。
- [ ] **ログアウト時のセッション破棄**: ログアウト処理の実行後、セッションが破棄され、ブラウザの「戻る」ボタンを押しても保護された画面での操作が行えなくなるか。
- [ ] **エラー画面の情報保護**: システムエラーや404エラーが発生した際、スタックトレースやデータベース接続文字列などの内部情報がエラー画面に表示されないようになっているか。本番環境では詳細なエラーログをサーバー側に出力し、画面には安全なエラーメッセージのみを表示する設計になっているか。
- [ ] **パスワードの秘匿**: ユーザーのパスワードが画面上に平文で表示されたり、ログファイルやGitHubのコミット履歴に平文で記録されたりしていないか。

## 4. 個人情報・デモデータの確認
デモデータには、実在人物の個人情報ではなく、確認用の架空データを使用する方針です。氏名・メールアドレス・申請理由・通知内容に個人情報が含まれていないかを確認します。

- [ ] **架空の個人情報・メールアドレスの確認**: 登録されているユーザー名やアドレス（`@example.com` などのテスト用ドメイン）が架空のものであるか。
- [ ] **架空の電話番号・住所の確認**: データベースや画面上に、実在する個人の電話番号や住所が登録されていないか。
- [ ] **実在しない社員番号の確認**: 社員番号などの識別子に、個人を特定できる実際のコードが使用されていないか。
- [ ] **過去の業務・機密情報の排除**: 過去の職場、学校、顧客で扱っていた実際の人物名やプロジェクト名などのデータが残っていないか。
- [ ] **デモデータとしての適切性**: 申請理由（例：「体調不良のため」など）や通知文面に、個人的な事情や不適切な内容が含まれていないか。
- [ ] **データリセット機能の確認**: 必要に応じて、本番デモ環境のデータを初期状態にリセットする手順や環境変数が整っているか（手動確認項目）。

## 5. GitHub上の機密情報確認
公開リポジトリ（GitHub）に、認証情報などの機密情報が誤ってコミットされていないかを確認します。

- [ ] **環境変数ファイルの除外**: `.env` などの環境設定ファイルが `.gitignore` に指定されており、Gitの追跡対象から外れているか。
- [ ] **ソースコード内のパスワード排除**: `DB_PASSWORD` や各種APIキーなどの認証情報が、Javaコード内やREADME.mdに直接書き込まれていないか。
- [ ] **JDBC接続URLの秘匿**: Supabase等の本番データベースに接続するための `JDBC_URL` に、実値のパスワードが書き込まれていないか（環境変数経由でのみ読み込む設計になっているか）。
- [ ] **Supabase接続文字列の実値**: Supabaseの接続情報（プロジェクトURLやAPIキーなど）が、フロントエンドやバックエンドのソースコード上にコミットされていないか。
- [ ] **Renderの環境変数**: Renderの管理画面で設定する環境変数の実際の値が、ソースコード内のファイルに平文で書き込まれていないか。
- [ ] **初期管理者のログイン情報**: 初期セットアップで生成される人事ユーザーなどの本物のメールアドレスや初期パスワードが、コード上に残っていないか。
- [ ] **テストコード・ログの確認**: JUnit等のテスト用コードや、過去のログファイルの中に、本番環境のパスワードや接続情報が残っていないか。
- [ ] **スクリーンショットの確認**: READMEやドキュメントに貼り付けている画面キャプチャ画像に、機密情報や個人を特定できるブラウザのタブ・URLが写り込んでいないか。

## 6. 共有前の最終確認
共有直前に、提出する文面や情報に不足がないかを最終チェックします。

- [ ] **アプリURLの正確性**: 共有するWebアプリのURLが正しいか。
- [ ] **GitHubリポジトリURLの正確性**: 共有するGitHubリポジトリのURLが正しいか。また、リポジトリが「Public（公開）」に設定されているか。
- [ ] **デモアカウント情報の整理**: 
  - 人事（HR）、店長（MANAGER）、一般従業員（EMPLOYEE）それぞれのログインに必要なメールアドレスとパスワードが、分かりやすくテキスト等で整理されているか。
  - *※上記は例であり、実際のデモ用ログイン情報は共有時に別途記載します。*
- [ ] **READMEの構成確認**: アプリの目的、使用技術、起動方法、デモアカウント情報がREADME.mdに適切に記載されているか。
- [ ] **関連ドキュメントの導線**: 関連ドキュメントへのリンク（本チェックリスト含む）が、README.mdの「関連ドキュメント」一覧から確認できるか。
- [ ] **確認してほしい機能のアピール**: ポートフォリオとして工夫した点（ロールごとの権限制御、Supabase Connection Pool、勤怠・シフト調整の整合性など）が明確になっているか。
- [ ] **起動遅延の共有措置**: Render Freeプランを使用しているため初回アクセス時に起動まで時間がかかることがある旨を、READMEに記載しているか。
- [ ] **セキュリティ対策**: 必要に応じてデモユーザーのパスワードを変更したり、デモ環境を停止・削除したりする手順を理解しているか（手動確認項目）。
