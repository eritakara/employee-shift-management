# 開発プロセスにおける課題解決と工夫点

本システム（シフト・勤怠管理Webアプリ）の設計・開発過程において、技術的な難所をどのように解決したか、およびシステムの信頼性・セキュリティを高めるために行った工夫についてまとめます。

---

## 1. 開発過程における設計上の工夫

### ① マルチDBに対応する「SQL動的互換レイヤー」の自作
ローカル開発・テストの生産性を高めるための軽量な埋め込み型DB（H2）と、本番環境のフルマネージドDB（Supabase PostgreSQL）をスムーズに共存させるため、動的に SQL 構文を書き換える仕組みを自作しました。
* `Database.isPostgres()` メソッドを実装し、実行中のデータベースが PostgreSQL かどうかを判定。
* PostgreSQL 接続時には、クエリに含まれる `DATEADD('day', 1, work_date)` を `(work_date + INTERVAL '1 day')::date` に書き換えるなど、SQL 内の日時計算や関数の差異を Java のデータベース・サービス層で安全に置換・分岐するレイヤーを構築しました。
* これにより、プロダクションコードベースを一本化したまま、ローカル検証の容易性と本番環境の堅牢性を高い次元で両立させました。

### ② 本番環境のセキュリティ安全装置（防衛的プログラミング）
ポートフォリオとして一般公開するにあたり、設定漏れによるセキュリティ上のインシデントを防ぐための「起動ブロック安全装置」を Java の起動初期化処理（`Database.java`）に組み込みました。
* 環境変数 `DEMO_ACCOUNTS_ENABLED=true` の場合のみデモアカウントが自動生成される構成とし、不用意な本番データ投入を防ぎます。
* 初期人事管理者 (`hr@example.com`) やデモアカウントのパスワードが、デフォルトの `Password1!`（ダミー値）のままで本番デプロイされた場合、アプリは `IllegalStateException` を発生させて起動自体を強制停止します。
* ログイン画面や公開リポジトリのドキュメントから具体的なアカウント資格情報を一切排除したうえで、ハッシュ化（`PasswordUtil.hash`）して保存する安全な設計を担保しました。

### ③ 設計改善と問題発生時の主体的な意思決定
開発の各フェーズにおいて、単純な動作確認に留まらず、システムの保守性と品質を高めるためのアプローチを自発的に行いました。
* **動作確認とエラー原因の切り分け**: ローカル環境およびRender Logsのログ監視を徹底し、Supabaseでの接続失敗やデータインサート時のエラーをログから特定して解決しました。
* **保守性を高めるリファクタリング**: 密結合になっていたServletとDAOの間にドメインごとのサービス層（`ShiftService` や `AttendanceService`）を設け、共通処理を `DateUtil` や `ServletUtil` などのユーティリティクラスへ抽出するリファクタリング方針を自ら決定し、設計を改善しました。

## 2. 開発中に直面した技術的課題と解決策

### 課題1: H2 と PostgreSQL での関数・集計仕様の差異による SQL エラー
* **状況**: 開発環境の H2 では正常に動作していた勤怠集計処理やシフト調整判定が、本番の PostgreSQL 移行時に `DATEADD` や `FORMATDATETIME` の未定義エラーとなり動作しなくなりました。
* **解決方法**: 
  `Database.java` に `isPostgres()` 判定処理を追加し、`DashboardService` や `ShiftService` でクエリを投げる際に `Database.isPostgres()` の真偽値に基づいて SQL 文字列を置換・分岐させるロジックを実装。また、TIME 型バインドにおける JDBC の `setString` から `setTime` への変更等を行い、互換性を担保しました。

### 課題2: Render Free Plan 上での Tomcat の動作・ヘルスチェック失敗
* **状況**: Render のヘルスチェック時に Tomcat のシャットダウンポート (8005) のアクセスが拒否される、あるいはスリープ復帰時 (Free Plan) にポートバインドが競合してコンテナがクラッシュを繰り返す問題が発生しました。
* **解決方法**: 
  `Dockerfile` のビルドスクリプト内で、Tomcat の `server.xml` を書き換えてシャットダウンポートを完全に無効化（`-1` に設定）し、待ち受けポートを Render の期待する `10000` に合わせる構成を実装。コンテナ挙動を最適化することで、安定起動を実現しました。
