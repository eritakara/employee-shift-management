# 開発プロセスにおける課題解決と工夫点

このドキュメントでは、シフト・勤怠管理Webアプリを開発する中で直面した課題と、その解決のために工夫した点をまとめています。

---

## 1. 開発中に工夫した点

### ① 開発環境と本番環境のデータベース差を吸収する仕組み

このアプリでは、ローカル開発やテストでは軽く動かせる **H2 Database** を使い、本番環境ではクラウド上の **Supabase PostgreSQL** を使っています。

ただし、H2 と PostgreSQL では、同じSQLでも書き方が少し異なる場合があります。そのため、環境によってSQLの違いを自動で調整する仕組みを作りました。

* `Database.isPostgres()` メソッドで、現在使っているデータベースが PostgreSQL かどうかを判定しています。
* PostgreSQL 接続時には、H2 用の日時計算SQLを PostgreSQL 用の書き方に置き換えるようにしています。
* これにより、ローカル環境と本番環境で別々の処理を大きく分けずに、同じアプリとして動かせるようにしました。

この工夫により、開発中の確認をしやすくしながら、本番環境でも安定して動作する構成にしています。

### ② 本番環境での安全性を高める仕組み

本番環境でアプリを公開する際に、設定ミスによって危険な状態のまま動かないよう、安全確認の仕組みを入れています。

デモ用アカウントは、`DEMO_ACCOUNTS_ENABLED=true` が設定されている場合のみ自動作成されます。また、初期管理者アカウントやデモアカウントのパスワードが未設定・空文字・初期値のままの場合は、本番環境でアプリが起動しないようにしています。

ログイン画面やGitHub上には実際のアカウント情報やパスワードを載せず、必要な相手にだけ別途共有する方針です。パスワードは平文で保存せず、ランダムなソルトを使ったPBKDF2-HMAC-SHA256（600,000回）でハッシュ化して保存します。

従来の120,000回形式のハッシュも検証可能な状態を維持し、利用者が正しいパスワードでログインした時点で現在の形式へ自動更新します。この段階移行により、既存利用者へ一斉のパスワード再設定を要求せずに保存方式を強化します。

これにより、デモとして確認しやすい状態を保ちながら、公開時の安全性にも配慮しています。

### ③ 問題発生時の原因調査と設計改善

エラーが起きた場合に原因を確認し、今後も修正しやすい構成になるように見直しました。

* ローカル環境や Render Logs のログを確認し、Supabase への接続失敗やデータ登録時のエラー原因を特定しました。
* Servlet と DAO に処理が集中しすぎないよう、`ShiftService` や `AttendanceService` などのサービス層に処理を分けました。
* 日付処理や画面遷移など、複数の場所で使う処理は `DateUtil` や `ServletUtil` にまとめました。

これにより、同じ処理の重複を減らし、あとから修正や確認をしやすい構成にしています。

## 2. 開発中に直面した課題と解決方法

### 課題1: H2 と PostgreSQL のSQLの違いによるエラー

**状況**  
ローカル環境の H2 では問題なく動いていた勤怠集計やシフト調整の処理が、本番環境の PostgreSQL では一部エラーになりました。

主な原因は、H2 と PostgreSQL で日付計算や時間の扱い方が異なることでした。

**解決方法**  
`Database.java` に PostgreSQL かどうかを判定する処理を追加しました。

そのうえで、`DashboardService` や `ShiftService` からSQLを実行する際に、利用中のデータベースに合わせてSQLの書き方を切り替えるようにしました。

また、時間データの登録方法も見直し、PostgreSQL で正しく扱える形に修正しました。

これにより、ローカル環境と本番環境のどちらでも同じ機能を安定して動かせるようにしました。

### 課題2: Render Free Plan 上での Tomcat 起動不具合

**状況**  
Render の無料プランでは、一定時間アクセスがないとアプリが一時停止します。その後、再びアクセスされたときに Tomcat が起動します。

この起動時に、Tomcat のポート設定やヘルスチェックの影響で、コンテナが正常に起動できない問題が発生しました。

**解決方法**  
`Dockerfile` のビルド処理の中で、Tomcat の設定ファイルである `server.xml` を調整しました。

具体的には、Tomcat のシャットダウンポートを無効化し、Render が期待する `10000` 番ポートでアプリが起動するように設定しました。

これにより、Render 上でもアプリが安定して起動し、ポート競合によるクラッシュを防げるようになりました。

## 3. セキュリティ調査と公開前の改善対応

ポートフォリオとして公開する前に、認証、権限管理、セッション、入力内容の確認、ログ出力などを見直しました。機能が動くだけでなく、誤った設定や不正な操作によって、必要以上の情報が見えたり変更されたりしないことを重視しています。

### 認証と連続操作への対策

* 本番環境では、初期管理者のメールアドレスやパスワードが未設定、空、または既定値のままの場合に起動を止めるようにしました。
* ログイン失敗とパスワード再設定要求には、接続元やメールアドレスに応じた回数制限を設けました。
* 回数制限の記録は一定時間で削除し、保存件数にも上限を設けることで、記録が増え続けないようにしています。

### 権限と表示情報の見直し

* 店長が閲覧できる有休申請と履歴は、原則として自分と同じ拠点・部署の従業員に限定しました。人事は、業務上必要な全体情報を確認できます。
* 他拠点のシフト表は、欠勤時などのヘルプ調整に必要なため、一般従業員と店長も閲覧できる仕様を維持しました。
* その一方で、閲覧用のシフト表には社員番号、詳細メモ、未確定シフトを表示せず、確定済みの勤務情報だけを確認できるようにしました。閲覧画面から他人や他拠点のシフトを変更することもできません。

### 入力内容とセッションの保護

* 勤怠修正申請では、出勤・退勤時刻の前後関係、対象勤務日との対応、24時間を超える勤務、理由欄の長さ、未処理申請の重複を確認するようにしました。
* セッションを識別するCookieには、画面上のスクリプトから読み取られにくくする設定と、他サイトから送信されにくくする設定を追加しました。
* ログイン後の画面をブラウザに保存させない設定を行い、本番のHTTPS通信では、次回以降も安全な通信を使うようブラウザへ伝える設定を追加しました。

### 画面と本番ログの情報保護

* エラーが起きても、利用者の画面には安全な案内だけを表示し、プログラム内部の情報を表示しないようにしています。
* 本番ログには、固定の処理名とエラーの種類だけを記録し、パスワード、再設定トークン、データベース接続情報、詳しいエラー内容や処理経路を出さないようにしました。
* 開発環境では原因調査ができるよう、必要な例外情報を確認できる状態を維持しています。

これらの対応により、拠点間のヘルプ調整など業務上必要な使いやすさを残しながら、公開デモとして必要以上の情報を見せない構成を目指しました。
